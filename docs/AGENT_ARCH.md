# TripiAgent v1 Agent Architecture & UX Guardrails

This document defines the architectural guidelines and design boundaries for the TripiAgent travel assistant. All agents, developer tools, and code generators must strictly adhere to these rules when building or modifying features.

---

## 1. Core Agent Model
*   **One Runtime Agent, Dual Personas:** We use a single Gemini 2.5/3.5 Flash model instance containing two contextual personas:
    *   **Planner Persona:** Active during itinerary generation, day planning, and review. Ensures spatial and temporal constraint validation.
    *   **In-Trip Persona:** Active when the user is traveling (detected via coordinates, time, and date). Reacts dynamically to live weather, traffic, or route disruptions.
*   **Stateless Server-Side Routing:** The agent does not persist conversational state on the server. The complete message history and the active itinerary context are sent on each request.
*   **Dev-Tooling Separation:** Developer guards (e.g. checks against code bloat, file structures, linting, rules checking) run purely as static offline checks (Vitest/Playwright, ESLint, CI/CD pipeline). They must **never** be part of the runtime client agent.

---

## 2. Memory & State Integration
*   **Zustand as Single Source of Truth:** The local Zustand store ([tripStore.ts](file:///C:/TripiAgent/stores/tripStore.ts)) holds the active trip state. 
*   **Append-Only Planning (Race Mitigation):** To prevent race conditions and layout glitches, the agent acts as an append-only query/edit engine. The agent outputs structured JSON day plans, which are applied to the store via single-batch updates (e.g., a `replaceDayPlan` method).
*   **UI Freeze During Re-planning:** When the agent is formulating updates or replanning a day, edits (such as drag-and-drop or edit/delete buttons in [ItineraryCard.tsx](file:///C:/TripiAgent/components/ItineraryCard.tsx)) are frozen on the UI.
*   **State-Saving Caching Layer:** We maintain a caching layer around `/api/ai` (using KV/local storage) to cache the last generated day plans so that the app fails gracefully with a toast notification during offline or poor cellular connection periods.

---

## 3. Data & Tooling Constraints
*   **No Memorized Schedules:** The agent must never memorize or hallucinate ferry schedules, transit timetables, or complex ZTL borders. It must instead query local datasets or helper tools:
    *   **Milan ZTL Check:** Small helper rule validator (`checkMilanZTL(time, day)`) providing active constraints.
    *   **Lake Garda Ferry Schedule:** Stored in a static JSON file (`public/data/lake_garda_ferries_2026.json`) and exposed through `/api/ferries`. The agent is required to call this tool if a ferry route is mentioned.
*   **Curated Places & Live APIs:** The agent retrieves real-time POI recommendations via `/api/places` (backed by Google Places + curated local stubs) and weather via `/api/weather`.

---

## 4. Mobile PWA UX Pattern
*   **Unified Chat Interface:** One main chat panel ([ChatInterface.tsx](file:///C:/TripiAgent/components/ChatInterface.tsx)) with dynamic quick-prompts matching the state:
    *   *Planning Mode:* Prompts for itinerary creation, day-by-day suggestions, and pre-booking advice.
    *   *In-Trip Mode:* Prompts for local weather swaps, restaurant bookings, or emergency rerouting (e.g., "replan due to traffic").
*   **Interactive Before/After Review:** Itinerary edits generated by the agent must be reviewed in a "Before vs After" drawer interface. The store is not mutated until the user explicitly clicks "Confirm & Apply".
