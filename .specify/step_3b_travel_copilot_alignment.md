# Specification: Step 3b — Travel Copilot Alignment & Value-First Engine

This specification outlines the functional requirements to align the comprehensive features of the external "Travel Planner + AI Co-Pilot" prompt with the lightweight, user-value-focused, database-less architecture of **TripiAgent**.

## 1. Goal & Context
The goal is to adapt the heavy multi-agent features (LangGraph, Prisma/SQLite, Eventbrite/OpenTable, NextAuth) from the external prompt into a lightweight, value-driven execution engine in TripiAgent. 
*   **Target Audience:** Italy travelers using a mobile PWA.
*   **Core Value:** Provide proactive warnings, daily briefings, and spontaneous ideas (Serendipity) without overloading the user with unnecessary options, sign-ups, or micro-management.

## 2. User Stories
*   **As a traveler,** I want to see a simple **Morning Briefing** card at the top of my itinerary, so that I know what to expect and what adjustments are suggested for today.
*   **As a traveler,** I want to get one spontaneous, high-quality **Serendipity recommendation** (local secret, viewpoint, street food) per day, so that my trip feels local and authentic without overwhelming me with hundreds of generic choices.
*   **As a traveler,** I want the Co-Pilot to warn me of weather alerts (e.g. rain during an outdoor swim or walk) and offer an easy before/after swap to an indoor activity.

## 3. Functional Requirements

### 3.1 Proactive Co-Pilot Widgets
- **Morning Briefing Card:** Render a small, dismissible, text-based morning overview at the top of the Home or Itinerary dashboard. It summarizes weather conditions and lists any travel alerts (e.g. ZTL entries today, ferry status).
- **Serendipity Nudge:** A dedicated single recommendation card generated by the agent for the active day, offering a unique local tip or spot.
- **Weather Swap CTA:** If active weather forecasts indicate precipitation, display a alert banner on the affected activity with a button to find an indoor alternative.

### 3.2 Simplified Architecture Mapping (Value-First, No Data Overload)

| Heavy Prompt Requirement | TripiAgent Lightweight Mapping | Reason |
| :--- | :--- | :--- |
| **LangGraph + FastAPI** | Single-model Gemini 2.5 Flash with structured system instructions & local tools. | Avoids hosting costs and offline latency; simple serverless routes are faster and more reliable. |
| **Prisma + SQLite DB** | Zustand State Store + LocalStorage persistence. | Zero-config database setup, instant page loads, and native offline support for PWA. |
| **NextAuth.js** | Client-side user profile stored in Zustand. | Avoids user onboarding friction. Value is delivered immediately without log-in barriers. |
| **Eventbrite / OpenTable** | Hardcoded/curated local API stubs + static schedules. | Prevents API limit bottlenecks and generic, cluttered listings; curated choices provide higher travel value. |

## 4. UI & Form Factor Constraints
*   **Viewport:** Optimized for 390px mobile screens.
*   **Interaction:** Simple card alerts on the dashboard; tapping a widget opens the chat or drawer to apply modifications. No cluttered lists or massive tables.

## 5. Security & Edge Cases
*   **Precautionary Default:** If live weather or geolocation APIs fail, fallback gracefully to pre-cached itinerary details and general tips.
*   **Offline Mode:** Keep the briefings and serendipity cards stored locally in the Zustand state so they remain visible when the user has poor connectivity during travel.
